方法

  • 重载和重写有什么区别？

​ 重载就是同样的方法根据输入数据不同，做出不同的处理。java允许任何函数的重载。

​ 重写就是子类对父类方法的重新改造，方法名和参数相同（外部相同），但是内部逻辑可以完全不同。

  • 面向对象和面向过程的区别？

  • 创建对象用什么运算符？对象实体与对象引用有什么不同？

  • 对象相等和引用相等的区别？
    • 对象的相等是内存中存放的内容相等
    • 引用的相等是内存地址的相等
  • 接口和抽象类的共同点和区别？

共同点：

  1. 无法实例化
  2. 都可以包含抽象方法
  3. ……

区别：

  1. 接口对类进行行为约束，实现接口必有对应行为；抽象类强调代码复用，就是父类模版

  2. 只能单继承，但可以实现多个接口

  3. 接口中的成员变量只能是public static final，抽象类默认default

  • 深拷贝和浅拷贝区别了解吗？什么是引用拷贝？

    • 浅拷贝：会在堆上创建新对象（区别引用拷贝），如果被拷贝对象是引用类型，浅拷贝直接复制其引用地址。
    • 深拷贝：完全复制整个对象，包含对象所包含的内部对象。

    浅拷贝会因为原对象的内容改变，拷贝内容改变。而深拷贝不会，拷贝出来的对象是独立于原对象的。

    浅拷贝和引用拷贝都是用原来堆中的地址，只不过浅拷贝要新建对象。

Java集合概述

  • List、Set、Queue、Map四者的区别？
    • List（对付顺序的好手）：存储元素有序，可重复
    • Set（独一无二的特质）：存储元素无序，不可重复
    • Queue（队列）：先入先出的顺序，可重复
    • Map（kv存储）：存储元素无序，k不可以重复，v可以重复
  • 四者的数据结构如何？
    • List：
      • ArrayList：Object[] 数组
      • LinkedList：双向链表（JDK1.6以前为循环链表，JDK1.7取消了循环）
      • ……
    • Set：
      • HashSet（无序，唯一）：基于HashMap实现
      • LinkedHashSet：LinkedHashSet是HashSet的子类，基于LinkedHashMap实现
      • TreeSet（有序，唯一）：红黑树（自平衡的排序二叉树），可以自主排序的Set
    • Queue：
      • PriorityQueue：Object[] 数组实现二叉堆
      • ArrayQueue：Object[] 数组 + 双指针
    • Map：
      • HashMap：JDK1.8以前，数组+链表解决哈希冲突；JDK1.8以后，当链表长度大于阈值（默认8），变成红黑树降低时间复杂度，如果当前数组长度小于64，那么优先扩容数组，这样搜索的时间复杂度更低
      • LinkedHashMap：继承于HashMap，在HashMap的基础上增加了双向链表，使其可以保持插入时的顺序
      • TreeMap：红黑树（自平衡的排序二叉树），可以自主排序的Map

设计模式

装饰器模式：在不改变原有对象的情况下拓展功能（应对继承关系比较复杂的场景）。通过构造器嵌套对象。

工厂模式：工厂模式创建对象，根据不同的任务需求，通过工厂类返回不同的对象，这些对象实现了统一的接口。

IO模型详解

  • BIO

​ 同步阻塞IO模型中，应用程序发起read调用后，一直阻塞，直到内核把数据拷贝到用户空间

  • NIO

​ 支持面向缓冲基于通道的IO操作方法。对于高负载、高并发的（网络）应用，应使用NIO

​ 同步非阻塞和HTTP1.1很像他没有阻塞你顺序访问，但他还是顺序执行，减少了等待结束的时延。

IO 多路复用模型，通过减少无效的系统调用，减少了对 CPU 资源的消耗。

Java 中的 NIO ，有一个非常重要的选择器 ( Selector ) 的概念，也可以被称为 多路复用器。通过它，只需要一个线程便可以管理多个客户端连接。当客户端数据到了之后，才会为其服务。

【选择器会选择一个已经数据从内核准备好的线程发起read请求】找到那个能成事的线程。所以NIO不是同步非阻塞模型而是IO多路复用模型。

  • AIO

​ 异步 IO 是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会堵塞在那里，当后台处理完成，操作系统会通知相应的线程进行后续的操作。（哥们只是发出请求，然后就等你回应了）

并发编程

  • 什么是进程和线程？
    • 进程：程序的一次执行过程，是系统运行程序的基本单位，是资源分配的基本单位。
    • 线程：比进程更小的执行单位，是cpu调度的最小单位。

​ 总结：线程是进程划分出来的更小的运行单位。线程和进程最大的不同在于基本上各进程是独立的，而各线程则不一定，因为同一进程中的线程极有可能会互相影响。线程执行开销小，但不利于资源的管理和保护。

  • 并发和并行？
    • 并发：两个及两个以上的作业在同一 时间段 内执行
    • 并行：两个及两个以上的作业在同一 时刻 执行
  • 同步和异步的区别？
    • 同步：发出一个调用请求后，没有得到响应之前不可以返回，一直等待。
    • 异步：发出一个调用请求后，不用等待返回结果，该调用直接返回。
  • 死锁是什么？如何预防和避免线程死锁？
    • 死锁四大条件：互斥条件，请求并保持条件，不可剥夺条件，循环等待条件（两个及以上的线程首尾相接的循环等待资源关系）
    • 如何预防死锁：
      • 破坏请求并保持条件：一次性申请所有资源
      • 破坏不可剥夺条件：如果申请不到资源就放手自己已有的共享资源
      • 破坏循环等待条件：按序申请资源，其实银行家算法也是这么做的
    • 如何避免线程死锁：
      • 银行家算法：在分配前提前计算好待分配的资源
  • volatitle 可以保证原子性嘛？

​ 不可以。只能保证可见性，原子性可以由synchronized、AtomicInteger和ReentrantLock改进

  • 乐观锁和悲观锁？
    • 乐观锁：默认好情况，出现错误才回滚。比如mysql中的mvcc机制。乐观锁通常多于写比较少的情况下（多读场景），避免频繁加锁影响性能，大大提升了系统的吞吐量
    • 悲观锁：默认会出问题，所以先上锁再操作悲观锁通常多用于写多比较多的情况下（多写场景），避免频繁失败和重试影响性能。
  • CAS是什么了解吗？

​ CAS 的思想很简单，就是用一个预期值和要更新的变量值进行比较，两值相等才会进行更新。当多个线程同时使用 CAS 操作一个变量时，只有一个会胜出，并成功更新，其余均会失败，但失败的线程并不会被挂起，仅是被告知失败，并且允许再次尝试，当然也允许失败的线程放弃操作。

  • synchronized 关键字？

​ synchronized 是 Java 中的一个关键字，翻译成中文是同步的意思，主要解决的是多个线程之间访问资源的同步性，可以保证被它修饰的方法或者代码块在任意时刻只能有一个线程执行。jdk1.6以后已经进行了优化，已经是轻量级的锁了。

​ 他可以修饰3个地方：实例方法（class类上锁）、静态方法（class类上锁）、代码块（对象上锁）

  • ReentrantLock 关键字？

​ ReentrantLock 实现了 Lock 接口，是一个可重入且独占式的锁，和 synchronized 关键字类似。不过，ReentrantLock 更灵活、更强大，增加了轮询、超时、中断、公平锁（按排队顺序）和非公平锁（不按排队顺序）等高级功能。

​

  • synchronized 和 ReentrantLock 异同？
    • 相同：都是可重入锁，线程可以获取自己内部的锁
    • 不同：后者提供，等待可中断（等待的线程可以选择放弃等待）；可实现公平锁（先等待的先得锁）；实现选择性通知
  • ThreadLocal 有什么用？

​ 实现每一个线程都有自己的专属本地变量

  • 线程池的核心参数？
    • corePoolSize : 核心线程数线程数定义了最小可以同时运行的线程数量（最小线程）。
    • maximumPoolSize : 当队列中存放的任务达到队列容量的时候，当前可以同时运行的线程数量变为最大线程数。（队列上限后，最大线程）
    • workQueue: 当新任务来的时候会先判断当前运行的线程数量是否达到核心线程数，如果达到的话，新任务就会被存放在队列中。（缓冲长度）

垃圾回收机制

当 JVM 无法为一个新的对象分配空间时会触发 Minor GC（每次 Minor GC 会清理年轻代的内存）、Major GC 是清理老年代、Full GC 是清理整个堆空间—包括年轻代和老年代

总结：

针对 HotSpot VM 的实现，它里面的 GC 其实准确分类只有两大种：

部分收集 (Partial GC)：

  1. 新生代收集（Minor GC / Young GC）：只对新生代进行垃圾收集；

  2. 老年代收集（Major GC / Old GC）：只对老年代进行垃圾收集。需要注意的是 Major GC 在有的语境中也用于指代整堆收集；

  3. 混合收集（Mixed GC）：对整个新生代和部分老年代进行垃圾收集。

整堆收集 (Full GC)：收集整个 Java 堆和方法区。

【为了更好的GC我们要先分批次也就是分区】

  • 何时进入老年代？

​ 大对象就是需要大量连续内存空间的对象（比如：字符串、数组）

​ 长期存活的对象进入老年代：虚拟机采用了分代收集的思想管理内存，先要做到如何识别年代。

​ 年轻代（初始为1）通过每一次minor gc涨一岁直到经过一个阈值（默认15岁），会晋升为老年代。

​

  • 空间分配担保是什么？

​ 确保在Minor GC前老年代的内存区域还有可以容纳新生代所有对象的剩余空间

【那么GC之前我们要判断谁要被GC】

  • 死亡对象判断方法？

    • 引用计数法（当对象相互引用的时候就永远不能为0）：

      1. 每当有一个地方引用它，计数器+1
      2. 引用失效，计数器-1
      3. 任何时候计数器为0的对象就是不可能再被使用的

    • 可达性分析算法

      找GC Roots，回溯找引用链，如果找不到代表此对象不可用，将被回收

      • 那么谁可以成为GC Roots呢？
        • 虚拟机栈（栈帧中的本地变量表）中引用的对象
        • 本地方法栈（Native 方法）中引用的对象
        • 方法区中引用的对象
        • ……
  • 引用类型总结？
    • 强引用：普遍的引用，垃圾回收器不会回收，内存空间不足就报错
    • 软引用：空间不足时，垃圾回收器会回收
    • 弱引用：比软引用更短暂，只要垃圾回收器扫描到他无论空间足够无法都回收
    • 虚引用：任何时候都可能被垃圾回收

【那么我们既然知道谁要被回收了，就看看我们如何回收，垃圾收集的算法】

  • 垃圾收集算法？

    • 标记-清除算法

      分为 “标记” 和 ”清除“ 阶段：先标记不需要清楚的，然后统一回收没有标记的，后面都是对其改进。两个问题：1、效率差；2、产生大量不连续的内存碎片

    • 标记-复制算法

      解决效率问题，我们不做标记和删除，直接讲内存分两半，将标记的对象排列好放入另一半，然后清空前一半。

    • 标记-整理算法

      根据老年代特点设定的算法，让存活的向一端移动，清掉端边界以外的内存

    • 分代收集算法

      分为新生代和老年代

      • 新生代每次收集都有大量对象死去，选择“标记-复制”算法，少量复制的成本就可以完成GC
      • 老年代存活几率高，没有额外空间担保，所以必须选择 标记-清除 or 标记-整理 算法

​ 题外话：为何要区分新生代和老年代呢？因为不同的对象在不同的场景下存活时间不同，要有针对性根据不同的GC算法来处理

【那么谁来执行这些算法呢？垃圾收集器】

  • 垃圾收集器？
    • Serial 收集器：如他的名字串行收集器，单线程的。新生代用标记-复制，老年代用标记-整理。优点：简单高效
    • ParNew 收集器：Serial收集器的多线程版本，和CMS结合紧密。首选
    • Parallel Scavenge 收集器：更关注吞吐量的ParNew收集器，CMS更在乎更短的停顿时间GC（提高用户体验）
    • Serial Old 收集器：老版
    • Parallel Old 收集器：老版
    • 🌟CMS收集器：以获取最短回收停顿时间为目标的收集器，垃圾收集线程与用户线程（基本上）同时工作，因为用标记-复制，所以缺点很明显，有内存碎片。
    • G1收集器：面向服务器的GC，满足停顿时间+高吞吐
    • ZGC收集器

  3. 栈（线程独享）
    代码的运行空间，里面存储我们编写的方法

  4. 程序计数器（线程独享）

    记录线程执行到哪里，为上下文切换服务

  5. 本地方法区（线程独享）
    不太重要，都是c写的native关键字那个，我们一般不涉及

【小总结】

  1. Java 文件经过编译后变成 .class 字节码文件
  2. 字节码文件通过类加载器被搬运到 JVM 虚拟机中
  3. 虚拟机主要的 5 大块：方法区，堆都为线程共享区域，有线程安全问题，栈和本地方法栈和计数器都是独享区域，不存在线程安全问题，而 JVM 的调优主要就是围绕堆，栈两大块进行

其实也不用管太多，只需要知道对象实例初始化时会去方法区中找类信息，完成后再到栈那里去运行方法。找方法就在方法表中找

  • 类加载器选择与双亲委派机制？

​ 当一个类收到了加载请求时，它是不会先自己去尝试加载的，而是委派给父类去完成，只有当父类加载器都反馈自己无法完成这个请求（也就是父类加载器都没有找到加载所需的 Class）时，子类加载器才会自行尝试加载。这样保证了不同的类加载器得到的都是同一个结 果，也帮助后面jvm比较两个类的时候，不但比较二进制的名字相同与否还比较是否是同一个类加载器

【总结JVM整体架构】

JVM 主要由四大部分组成：

  • ClassLoader（类加载器）：将.class文件加载，在jvm中生成对象
  • Runtime Data Area（运行时数据区，内存分区）：线程私有（执行），虚拟机栈，本地方法栈，程序计数器；线程共享（公共信息），堆，方法区
  • Execution Engine（执行引擎）：将 JVM 指令集翻译为操作系统指令集
  • Native Interface（本地库接口）：他的作用是调用不同语言的接口给 JAVA 用

调优主要在堆和栈中进行，在堆中进行内存大小or新生代老年代比例的调整；在栈中进行参数的调整。

  • 什么时候选TCP，什么时候选UDP？
    • UDP用于即时通信，刷视频
    • TCP用于传输准确性高的场景，文件传输
  • HTTP是基于TCP还是UDP呢？

​ 准确来说HTTP3.0之前都是基于tcp连接的，http3.0适用基于UDP的QUIC协议，可以用来解决HTTP/2的队头阻塞问题。由于HTTP/2在单个TCP连接上面使用了多路复用，但是如果出现了丢包，整个连接都会阻塞（http1.x都是不丢包，单纯没处理完就会阻塞）。

  • TCP三次握手？为什么？

​

第一次握手：客户端发送SYN数据包给服务端，客户端进入SYN_SEND状态，等待服务器的确认

第二次握手：服务端发送SYN + ACK数据包给客户端，服务端进入SYN_RECV状态

第三次握手：客户端发送ACK数据包给服务端，双端进入ESTABLISHED状态，完成TCP三握手

客户端 服务端
第一次握手 什么都确认不了 确认客户端发送正常，自己接收正常
第二次握手 确认自己发送接收都正常、确认对方发送接收正常 确认客户端发送正常，自己接受正常
第三次握手 确认自己发送接收都正常、确认对方发送接收正常 确认自己发送接收都正常、确认对方发送接收正常

三次握手就是确认双方都收发正常的过程，两次握手不能保证服务端的发送和接收端的接收正常

  • 四次挥手？为什么要四次？

​

第一次挥手：客户端发送一个FIN数据包，用来关闭客户端到服务端的数据传输

第二次挥手：服务端收到FIN数据包发出ACK数据包，应答了客户端的FIN数据包也就是终止请求

第三次挥手：客户端发送一个FIN数据包，用来关闭客户端到服务端的数据接收

第四次挥手：客户端收到FIN数据包发出ACK数据包，应答了服务端的FIN数据包也就是终止请求

为什么要四次呢？因为前两次挥手只是表示客户端不再发送信息，并不代表服务端已经完成接收，所以等服务端接收完成后，需要通过第三次和第四次的握手来关闭双方的通信

  • TCP如何保证传输的可靠性？
    • 基于数据块传输：给报文分段，不会出现语义混乱
    • 对失序数据包重新排序以及去重：标号保证数据包都按出发顺序到达
    • 校验和：TCP保持它首部和数据的校验和。检测数据在传输过程中没有变化。有问题就丢弃该数据包
    • 超时重传：如果在合理的往返时延（RTT）内未收到确认消息，则标记为丢失
    • 流量控制：通过滑动窗口控制发送端的发送流量，并通过数据包中的信息实时变动
    • 拥塞控制：网络阻塞时，减少数据的发送
  • ARQ协议是什么？

​ 自动重传请求（Automatic Repeat-reQuest，ARQ）是 OSI 模型中数据链路层和传输层的错误纠正协议之一。

ARP

  • 什么是MAC地址？

​ 就是一切网络设备上唯一标识的物理地址。对比IP是网络中的虚拟地址，谁都可以获得同一个IP，但不能同时存在同样的IP。

  • ARP协议是什么？

​ ARP全称是地址解析协议，用来将IP地址转化为MAC地址，具体工作在TCP/IP协议中的IP层，OSI模型的数据链路层。

工作原理：

    1. 同一局域网（子网）内的MAC寻址（广播问询，单播响应来更新ARP表）
    1. 跨局域网（子网）的MAC寻址（通过目标子网的网关来进行广播问询，单播响应更新ARP表）
